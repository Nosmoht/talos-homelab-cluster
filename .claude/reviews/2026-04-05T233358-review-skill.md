## Recommendations

### 1. Helm-values ServiceMonitor discovery is undefined (Impact: High, Category: Completeness)
**Evidence:** Phase 8, line 226: "If a `ServiceMonitor` resource was discovered in Phase 1 (via `resources/` YAML **or known to be in Helm chart values**)". Phase 1 only scans `resources/*.yaml`. Tetragon's ServiceMonitor is declared in `values.yaml` as `serviceMonitor.enabled: true`, not as a standalone resource file. Without a values-scan step, Phase 1 never records it and Phase 8 will SKIP the ServiceMonitor probe for Tetragon.

**Why it matters:** The reference component (Tetragon) would produce a false SKIP. Any Helm chart that bakes observability into its values has the same gap.

**Validation:** Re-run against Tetragon: Phase 8 should probe `ServiceMonitor/tetragon` in namespace `tetragon`.

**Current:**
```
3. **Resources directory** — list all YAML files in `${OVERLAY_PATH}/resources/` (if it exists)
```

**Recommended:**
```
3. **Resources directory** — list all YAML files in `${OVERLAY_PATH}/resources/` (if it exists)
4. **Helm values scan** — if `${OVERLAY_PATH}/values.yaml` or `kubernetes/base/infrastructure/${ARGUMENTS}/values.yaml` exists, scan for:
   - `serviceMonitor.enabled: true` or `serviceMonitor.create: true` → add ServiceMonitor to inventory (name defaults to component name)
   - `prometheusRule.enabled: true` → add PrometheusRule to inventory
   - `grafana.dashboards` or `dashboardProviders` → add dashboard to inventory
   This detection supplements `resources/` YAML discovery — do not replace it.
```

**Current:**

Example for TracingPolicy (Tetragon): `cilium.io/v1alpha1 / TracingPolicy` — no rich status subresource...

**Recommended:**
```
Status subresource heuristics (apply to any custom CR):
- If `apiVersion` contains `v1alpha1` or `v1beta1`: status subresource is likely absent or sparse. Presence of the instance + ArgoCD `Synced` = OK. Note in output: "no status subresource".
- If the CR's `.status` object contains a `conditions` array: assert at least one condition with `type: Ready` or `type: Available` and `status: "True"`.
- If `.status` is empty or absent: treat as OK with note. Do not CRIT on missing status for alpha-API resources.
```

### 3. CRD-missing error not distinguished from instance-missing error (Impact: Medium, Category: Completeness)
**Evidence:** Phase 6: `resources_list: apiVersion=<apiVersion>, kind=<kind>, namespace=<ns>`. If the CRD is not installed, `resources_list` returns an API error ("no matches for kind X"), not an empty list. No guidance on handling this.

**Why it matters:** On a fresh cluster or after failed sync, the CRD might not exist. Reporting CRIT with wrong cause ("instance missing" vs "CRD not installed") misdirects debugging.

**Validation:** The skill should emit CRIT "CRD not installed" with `kubectl get crd` suggestion when the API server returns a kind-not-found error.

**Current:**
```
Assert:
- The instance named in the resource file is present → OK; missing → CRIT
```

**Recommended:**
```
Assert:
- If `resources_list` returns an API error containing "no matches for kind" or "resource type not found": CRIT "CRD not installed for <kind> — check `kubectl get crd | grep <group>`"
- If `resources_list` returns empty list (CRD exists, no instances): CRIT "Expected instance <name> not found"
- The instance named in the resource file is present → OK
```
