# Hubble Certificate Rotation

## Background

Hubble uses mTLS between cilium-agent (server), hubble-relay (client), and optionally hubble-ui (client).
The certificate secrets live in `kube-system`:

| Secret | Used by |
|--------|---------|
| `hubble-server-certs` | cilium-agent Hubble gRPC listener |
| `hubble-relay-client-certs` | hubble-relay → cilium-agent mTLS |
| `hubble-relay-server-certs` | hubble-relay gRPC server |
| `hubble-metrics-server-certs` | Hubble metrics TLS endpoint |
| `hubble-ui-client-certs` | hubble-ui → relay |

Certificates are generated by the `hubble-generate-certs` CronJob (schedule: `0 0 1 */4 *` — every
4 months, in `kube-system`). The bootstrap manifest `kubernetes/bootstrap/cilium/cilium.yaml` also
contains a one-shot Job with the same certgen image that ran at initial install.

**Key constraint from AGENTS.md §Cilium Bootstrap/Talos Nuance:**
`kubernetes/bootstrap/cilium/cilium.yaml` is tied to Talos `extraManifests`. Never `kubectl apply`
it to fix drift — always reconcile via `make -C talos upgrade-k8s`. Apply Talos configs to all CP
nodes **before** `upgrade-k8s` (reads `extraManifests` URL from live node config). `extraManifests`
does NOT garbage-collect — manually delete orphaned Jobs after bootstrap re-render.

## Environment Setup

Read `cluster.yaml` for cluster-specific values (node IPs, kubeconfig path).

## Rotation Paths

**Path A (Normal — preferred):** Trigger the CronJob manually by creating a new Job. No changes to
git or bootstrap files. Use for routine pre-expiry rotation when the cluster is healthy.

**Path B (Bootstrap re-render):** Re-render `cilium.yaml` via `make -C talos cilium-bootstrap` and
reconcile via `make -C talos upgrade-k8s`. Use when Cilium is being upgraded or when the certgen
image/schedule itself needs updating.

`--check-only`: run Step 1 only (expiry inspection), no rotation.

---

## Step 1 — Check Certificate Expiry

Inspect all Hubble cert secrets. The MCP server is read-only and returns base64 values; openssl
parsing requires kubectl CLI:

```bash
KUBECONFIG=<kubeconfig>
for secret in hubble-server-certs hubble-relay-client-certs \
              hubble-metrics-server-certs hubble-ui-client-certs; do
  echo "=== $secret ==="
  # Try tls.crt first, then ca.crt
  kubectl get secret "$secret" -n kube-system -o jsonpath='{.data.tls\.crt}' 2>/dev/null \
    | base64 -d | openssl x509 -noout -enddate 2>/dev/null \
  || kubectl get secret "$secret" -n kube-system -o jsonpath='{.data.ca\.crt}' 2>/dev/null \
    | base64 -d | openssl x509 -noout -enddate 2>/dev/null \
  || echo "  (secret not found or no recognized cert key)"
done
```

Use MCP for metadata (creation timestamp, labels — not cert data extraction):

```
resources_get(apiVersion="v1", kind="Secret", name="hubble-server-certs", namespace="kube-system")
# Check .metadata.creationTimestamp
```

**Rotation threshold:** Rotate if any cert expires within 30 days, or immediately if already expired.

If `--check-only`: report expiry dates and stop here.

## Step 2 — Verify CronJob Health

```
resources_get(apiVersion="batch/v1", kind="CronJob", name="hubble-generate-certs", namespace="kube-system")
# Check: .spec.schedule, .spec.suspend, .status.lastScheduleTime, .status.active
# Fallback: kubectl get cronjob hubble-generate-certs -n kube-system -o yaml
```

If CronJob is missing (`404`) or `.spec.suspend == true`: proceed with Path B.
If CronJob is healthy: proceed with Path A.

## Path A — Trigger CronJob Manually

### A1. Create a Job from the CronJob

```bash
KUBECONFIG=<kubeconfig> kubectl create job hubble-cert-rotate-manual \
  --from=cronjob/hubble-generate-certs \
  -n kube-system
```

### A2. Wait for Job completion

Poll up to 12 × 10 s = 2 min:

```bash
for i in $(seq 1 12); do
  phase=$(KUBECONFIG=<kubeconfig> kubectl get job hubble-cert-rotate-manual -n kube-system \
    -o jsonpath='{.status.conditions[?(@.type=="Complete")].status}' 2>/dev/null)
  failed=$(KUBECONFIG=<kubeconfig> kubectl get job hubble-cert-rotate-manual -n kube-system \
    -o jsonpath='{.status.failed}' 2>/dev/null)
  echo "$i: complete=$phase failed=$failed"
  [ "$phase" = "True" ] && { echo "Job complete"; break; }
  [ "${failed:-0}" -gt "0" ] && { echo "Job failed — check logs (Step A2b)"; break; }
  sleep 10
done
```

### A2b. On failure — check certgen logs

```bash
KUBECONFIG=<kubeconfig> kubectl logs job/hubble-cert-rotate-manual -n kube-system
```

Common failure: RBAC error (missing verb on hubble-generate-certs Role), or existing secret has
wrong ownership annotation. Report the error and escalate to Path B if unresolvable.

### A3. Verify cert expiry updated

Re-run the Step 1 expiry check. New certs from `cilium-certgen` default to a 3-year validity.
Confirm `notAfter` is significantly in the future (> today + 30 days).

### A4. Rolling restart of affected components (if needed)

Cilium agent picks up updated cert secrets automatically via projected volume re-read. If Hubble
observe shows TLS errors after A3, force a rolling restart:

```bash
KUBECONFIG=<kubeconfig> kubectl rollout restart daemonset cilium -n kube-system
KUBECONFIG=<kubeconfig> kubectl rollout status daemonset cilium -n kube-system --timeout=300s

KUBECONFIG=<kubeconfig> kubectl rollout restart deployment hubble-relay -n kube-system
KUBECONFIG=<kubeconfig> kubectl rollout status deployment hubble-relay -n kube-system --timeout=120s
```

### A5. Verify Hubble connectivity

```bash
# hubble-relay pod is Ready
KUBECONFIG=<kubeconfig> kubectl get pods -n kube-system -l k8s-app=hubble-relay
# cilium-agent health
KUBECONFIG=<kubeconfig> kubectl exec -n kube-system ds/cilium -c cilium-agent -- \
  cilium-dbg status --brief
```

### A6. Cleanup manual Job

```bash
KUBECONFIG=<kubeconfig> kubectl delete job hubble-cert-rotate-manual -n kube-system
```

## Path B — Bootstrap Re-render

Use when Cilium is being upgraded, the CronJob is missing/suspended, or the certgen image itself
needs updating. This path modifies the bootstrap manifest and reconciles via Talos.

### B1. Pre-check — no conflicting Jobs running

```bash
KUBECONFIG=<kubeconfig> kubectl get jobs -n kube-system -l k8s-app=hubble-generate-certs
```

If an existing Job is in `Running` state, wait for it to complete before proceeding.

### B2. Re-render bootstrap manifest

```bash
make -C talos cilium-bootstrap
```

This regenerates `kubernetes/bootstrap/cilium/cilium.yaml` from `CILIUM_VERSION` in `talos/versions.mk`.

Verify the manifest includes an updated `hubble-generate-certs` Job (name suffix changes per render):
```bash
grep 'name: hubble-generate-certs-' kubernetes/bootstrap/cilium/cilium.yaml
```

### B3. Validate — no static cert secrets in manifest

```bash
make -C talos cilium-bootstrap-check
```

If this fails (static `hubble-*-certs` Secret resources found in `cilium.yaml`): stop. Certs should
be generated dynamically by the Job, not baked into the bootstrap manifest. Investigate the Cilium
chart version.

### B4. Commit and push

```bash
git add kubernetes/bootstrap/cilium/cilium.yaml
git diff --cached --stat  # Confirm only cilium.yaml changed
git commit -m "chore(cilium): re-render bootstrap manifest for Hubble cert rotation"
git push
```

### B5. Apply Talos configs to all CP nodes first

`upgrade-k8s` reads `extraManifests` URLs from the **live** node config — apply updated configs
to all CP nodes before running `upgrade-k8s`:

```bash
make -C talos apply-node-01
make -C talos apply-node-02
make -C talos apply-node-03
```

Verify all CP nodes are `Ready` after apply:
```bash
KUBECONFIG=<kubeconfig> kubectl get nodes -l node-role.kubernetes.io/control-plane=
```

### B6. Reconcile via upgrade-k8s

```bash
make -C talos upgrade-k8s
```

This re-applies all `extraManifests` including updated `cilium.yaml`. A new `hubble-generate-certs`
Job is created (different hash suffix from the old one).

**extraManifests does NOT GC:** If the old Job still exists, two Jobs with different name suffixes
will both run certgen — this is idempotent but produces duplicate Jobs. Clean up old completed Jobs:

```bash
KUBECONFIG=<kubeconfig> kubectl get jobs -n kube-system -l k8s-app=hubble-generate-certs
KUBECONFIG=<kubeconfig> kubectl delete job <old-job-name> -n kube-system
```

### B7. Verify (follow Path A §A2–A5)

Poll for Job completion (A2), verify cert expiry (A3), restart components if needed (A4), and
verify Hubble connectivity (A5).

---

## Post-Rotation Checklist

- [ ] All cert secrets show `notAfter` > today + 30 days (Step 1 re-check)
- [ ] `hubble-generate-certs` Job shows `Completed`
- [ ] `hubble-relay` pod is `Running` and `Ready`
- [ ] `cilium-agent` DaemonSet pods show no TLS errors in logs
- [ ] `cilium-dbg status --brief` shows no TLS-related warnings
- [ ] Manual Job or old bootstrap Jobs deleted
- [ ] (Path B only) `kubernetes/bootstrap/cilium/cilium.yaml` committed and pushed

## Hard Rules

- Never `kubectl apply kubernetes/bootstrap/cilium/cilium.yaml` directly — use `make -C talos upgrade-k8s` only.
- Apply Talos configs to all CP nodes **before** `upgrade-k8s` — `upgrade-k8s` reads `extraManifests` URLs from live node config.
- `upgrade-k8s` does not garbage-collect — manually delete orphaned Jobs after Path B.
- After certgen Job completes, always verify the cert expiry dates — a Job can complete without
  renewing certs (e.g., if certgen skips existing unexpired secrets). Trust `openssl x509 -enddate`,
  not just Job status.
- Never force-delete the `hubble-generate-certs` CronJob — it is managed by `extraManifests` and
  will reappear on the next `upgrade-k8s` run. Suspend it instead if needed: `kubectl patch cronjob
  hubble-generate-certs -n kube-system -p '{"spec":{"suspend":true}}'`.
